Purpose

This WWW site will document the collaboration service simulation projects requested in support of Run2 analyses. The initial emphasis will be the support of presentations to be given at QM'02 in July, 2002.

The coordination effort for these simulation projects has been described on the phenix-p-l list. See the first related link below. Simulation projects are to be requested by one or more Physics Working Groups (PWGs), not by individuals nor by institutions. Individuals or institutions should make their requests to the one or more PWGs which have a close interest in the project. Upon discussion in the PWG meetings, the PWG conveners will designate a contact person for the simulation project and a relative priority for this project in terms of other projects requested by the PWG. The contact person will then advise me of the project, which I will post on this WWW site for collaboration review.

Simulation Project Management

Each coordinated simulation project is assigned a Project Number. Within each project number there are assigned unique Run Number ranges for the input and the output files. The input and output file number spaces are independent. A file of a given type (event input, PISA hits output, simulated PRDF, simulated DST, etc.) is uniquely identified by its Run Number. Before a simulation project begins production, the run number spaces are assigned so that other projects will not have overlapping run numbers for their files. All of the files information is to be stored in an Objectivity database.

Each coordinated simulation project is also assigned its own CVS tag before the start of production. The naming convention for the tags used in Run2 PISA hits files production is simProjectxx where xx is the assigned Project Number. In turn for each simulation project there is a dedicated subdirectory simulation/pisa2000/simRun2/projectx where the project contact persons can place particular input files related to their projects. Once these subdirectories have been filled, the CVS tag is made for the project, and the project is ready for files production.

The overall priority schedule for the simulations projects will be set at the weekly (Wednesday) analysis meetings, or during special core week meetings, or by the weekly PWGs conveners's teleconference. My own (CFM) role is primarily to be the coordinator, not the gatekeeper. Naturally, once a project is underway at a given farm site, it is most efficient to allow the project to conclude or to finish a well-defined milestone before interrupting the project. Similarly, if there is room on available computing farms, there is no reason to delay pending projects. The goal is to produce output efficiently, not to make people jump through hoops.
