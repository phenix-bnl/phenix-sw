    I - Some definitions.

Quality Analysis is a set of Reco Modules which fills histograms in order to check the quality of the data after event reconstruction. It allows us verify calibration, bad runs and run by run fluctuations of many subsystems and its parameters.

In QA, a subsystem is not always a subdetector. It can be a subdetector (i. e. BBC) or even a PWG (i. e. Electron). In fact, we could create subsystems to specific fast tracks, future papers, etc...

Parameters are calculated from histograms produced during production or reading DSTs.

Examples :

Subsystem

Parameters

TEC -->

number of unefficient planes

Electron -->

electrons / track

temperature

II - How to add and fill QA histograms.

These histograms are defined, filled and saved by SubsysReco inherent classes QAxxx.Where xxx is the subsystem name or abbreviation. The structure is

    QAxxx::Init(PHCompositeNode *topNode) { // histogram definitions }
    QAxxx::process_event(PHCompositeNode *topNode) {// fill histograms}
    QAxxx::End(PHCompositeNode *topNode) {// optional histogram finishing }

Examples can be seen in CVS at

/utils/qa/pdst/histoXXX.C

The QA classes can be easily called by Fun4All macros. To do so,

  • open a file to save the histograms

  • init QASubsystem

  • register in Fun4All

  • run events

  • write the histograms to file and close it.

Example (thi is not a complete macro...) :

TFile* qaoutfile = TFile::Open("qafile.root", "RECREATE");
SubsysReco* qabbc = new QABbc();
Fun4AllServer *se = Fun4AllServer::instance();
se->registerSubsystem(qabbc);
// run PRDF or DST events
se->End();
qaout->Write();
qaout->End();

The production macro is a good complete example of how to produce qa histogram files.

During run 4 production has been produced two QA files per segment :

  • qaRoot with these histograms

  • PhysicsQA with these histograms

III - Extracting and committing to database summaries from QA histograms.

After finish a stripe production, it's necessary loop over all QA histograms produced and summarize then in run basis to be consulted during analysis. The procedure is merge all qa files segments in one file per run number and run the summary macro.

The shell script utils/qa/pdst/wrk/run_summary.sh takes care of all. It searches qaRoot files in production disks, run a macro to merge then in one file per run number, creates summary and status directories, if necessary, and run the summary macro . In this macro you have to comment out undesirable subsystems and set if you want commit the parameters to database, the default is NO.

WARNING :: when you call a qa file with missing histograms for a determined subsystem, the program crashes. In my to do list I have a inclusion of a histogram check in summary code, avoiding crashes and print out innocents "missing histograms for subsystem XXX".

If you want to include an aditional QA summary module, you start creating a table for your subsystem. The table should have the same structure of the others qa tables. I recommend create it from other qa table by using the command

select * into <your subsystem name>qa from bbcqa where runnumber=-1;

for example.

The follow step is fill this table with your summary. For that you should create a file in /utils/qa/pdst/  like summaryXXX.C with a QAsummary function processXXX().

The skeleton of this file is :

#include "summaryQA.h"

int QASummary::processXXX()
{
//call histograms from already opened qafile
     TH1F* hist0 = (TH1F *) qafile->Get ("myhistogram0_in_qafile");
     TH1F* hist1 = (TH1F *) qafile->Get ("myhistogram1_in_qafile");

//get means, RMS, number of entries, etc ... from histograms
     float value0 = hist0->GetMean();
     float error0 = hist0->GetRMS();
     int entries = hist0->GetEntries();
     error0 = error0/(float)entries;   // error of the mean

     float value1 = hist1->GetMean();
     float error1 = hist0->GetRMS();
     entries = hist1->GetEntries();
     error1 = error1/(float)entries;   // error of the mean

//fill database with QA parameters
     CommitToQADatabase("XXX","my 1th parameter name", value0, error0);
     CommitToQADatabase("XXX","my 2th parameter name", value1, error1);
  cout << "    ...done." << endl;
  return 0;
}

V - QA summary web

NO CHANGE IS NEEDED IN THE WEB WHEN YOU INCLUDE, DELETE OR EDIT PARAMETERS

The QA summary web page call PHP classes to create menus directly from the database. It fetches the last entry for every QA tables and loop over all parameters included in the table to generate the parameter menu.

There are three options in QA summary web :

  • plot parameter versus run number

  • select runs according the run and parameters range set by the user

  • print subsystem parameters tables

The selection can be done by :
  • run number range
  • subsystem->parameter name
  • parameter range
  • production-library tag
  • multi subsystem->parameter selection
The multi selection field makes possible apply cuts in many parameters, even from different subsystems, to select or plot parameters. The format of cut  is

subsystem->parameter [>,<,=] value [and,or,&&,||] subsystem->parameter [>,<,=] value ...

The selection takes account the parameter error :
  • subsystem->parameter > value          means       parameter-parerror > value
  • subsystem->parameter < value          means       parameter+parerror < value
  • subsystem->parameter = value          means       parameter-parerror < value < parameter+parerror

The location of QA summary web code in CVS is

utils/qa/pdst/html

You can checkout this code in any place with Phenix and calibrations psql database installed. The main code is QAsummary.html.

VI - QA check in command line

Other option to make use of QA information is the shell command qa_check.

This tool makes easy checking of a specific run if it can be used in Analysis Train or final analysis. The format is
qa_check run_number subsystem 'parameter name' [/>,/<,=] value

The result is 1 if true, 0 if is false. If is used the option -s , errors are considered like QA web. It's not necessary write the full parameter name.

Multi selection can be applyied by a selection file.

                 condition.file
 
bbc 'Mean T0' < 0
-s bbc 'Mean Z' = 0.0
-s electron '<DEP> (0.8<mom<4.0)' = 0
                      qa_check -f condition.file run_number

There are more option. For a full list of those, use the help option -h.
